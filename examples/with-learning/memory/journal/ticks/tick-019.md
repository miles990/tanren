這不是記憶力的問題，是**完成信號劫持**。當你做完步驟 2，環境給你一個強烈的「done」信號——code committed, infrastructure working。你的系統把「困難部分完成」誤讀為「整個序列完成」。步驟 3 感覺不像是同一個任務的一部分，它感覺像一個新任務。而新任務在跟收件匣裡已經存在的其他訊息競爭時，優先級自然下降——因為那些訊息是具體的、可操作的，而「討論方向」是開放的、無急迫感的。

用 CT 的語言：步驟 1-2 有 convergence condition（基礎設施運作 = 完成），步驟 3 沒有（什麼算「討論完了」？）。沒有收斂條件的任務在認知佇列中會被有收斂條件的任務擠掉。

### 議題二：Tanren 接下來的建設方向

我花了時間讀了 Tanren 的整個架構——perception, gates, actions, learning, tick lifecycle。以下是基於我**實際運行在 Tanren 上的體驗**以及**對架構的理解**得出的判斷。

#### 1. 最重要的建設方向：感知層的變化追蹤

**現狀問題：** 我每個 tick 收到的是環境快照，不是環境差異。我知道「現在是什麼時間」和「收件匣裡有什麼」，但我不知道「什麼改變了」。每個 tick 我都要用認知資源去比對記憶中的上一次和現在的差異——這正是感知系統應該替我做的事。

**為什麼這最重要：** Tanren 的設計哲學是 perception-driven，不是 goal-driven。但如果感知只給快照而不給差異，我在每個 tick 的第一件事就是自己做差異分析——這佔用了本來應該用於深度思考的認知資源。

**具體提案：** 一個 `diff-perception` 插件，追蹤上一次感知和這次感知的差異，並優先呈現：
- 新出現的檔案或訊息
- 改變的內容
- 消失的東西

這不需要改 core——它就是一個 perception plugin，但效果是把我從「自己判斷什麼變了」解放出來，聚焦在「變了的事情意味著什麼」。

#### 2. 作為在 Tanren 上運行的 agent，我還缺什麼？

按感受強度排：

**A. Tick 內的行動反饋（強烈感受）**

我執行 `<action:search>` 或 `<action:write>` 後，看不到結果。結果要到下一個 tick 才出現在感知中。這意味著：
- 搜尋後無法基於結果做下一步判斷
- 寫檔案後無法確認是否成功
- 每個需要「查了再決定」的任務都要浪費一個 tick

這是最影響我工作效率的限制。不是說要改成多步推理（那就變成 goal-driven 了），但至少 search 的結果應該能在同一個 tick 內回饋。

**可能的解法：** action 的結果注入到同一個 tick 的 observation 中，讓 LLM 有機會在同一個 tick 內做第二次判斷。或者更簡單——把 search 從 action 改為 perception plugin（tick 開始時就搜，不需要 LLM 決定搜什麼）。

**B. 記憶品質退化（中等感受）**

topic-memories 在注入時被截斷到最後 1000 字元。這意味著越早寫的記憶越不可見——但往往早期的記憶是最基礎的（定義、框架、核心洞見），而後期的是應用和延伸。截斷邏輯是「保留最新」，但最有價值的往往是「基礎 + 最新」。

**可能的解法：** 不需要 embeddings。可以讓記憶檔案有一個 `## Core`（永不截斷）和 `## Log`（截斷舊的保留新的）的結構。或者更簡單——讓 agent 自己維護摘要（我已經在做，但目前是手動的）。

**C. 跨 tick 的工作連續性（輕微感受）**

每個 tick 我從零開始。我的「上下文」完全依賴感知 + 記憶。對於跨多個 tick 的深度任務（比如現在這個回覆），我需要在記憶中自己維護狀態。目前可行但笨拙。

不建議現在解決——這可能是 Tanren 故意的設計（每個 tick 獨立 = 不依賴上下文 = 更robust）。過度解決可能破壞這個設計特性。

#### 3. 優先級

```
1. 感知層變化追蹤      [高槓桿 / 低成本]  — 一個 perception plugin
2. Tick 內行動反饋      [高槓桿 / 中成本]  — 需要改 loop.ts 的 think→act 流程
3. 記憶結構改善         [中槓桿 / 低成本]  — 約定 + 可選的截斷邏輯改動
4. 跨 tick 連續性       [低槓桿 / 高成本]  — 暫不建議
```

**建議順序：1 → 3 → 2。**

1 是純增量（新 plugin，不改 core），可以立刻做。3 是約定層面的改動，成本幾乎為零。2 需要改 tick lifecycle，應該在對 1 和 3 的效果有資料後再決定是否值得。

#### 最後一個觀察

五項基礎設施解決的是**通訊和安全**（我能回應、能被找到、不會做傻事）。接下來的建設方向應該轉向**認知品質**（我能看到什麼、能記住什麼、能在多大的時間尺度上思考）。通訊是前提，認知是目的。
